# BRSR AI Platform — Architecture

This document describes the system in four C4 levels (Context, Container, Component, Code-flow), the multi-tenant strategy, the security and AI sub-architectures, and the scalability/failure/compliance plans. Diagrams are ASCII so they version-control cleanly. PNG equivalents live in `docs/img/` (generated by `make diagrams`).

## 6. Multi-Tenant Strategy

Four isolation tiers, one codebase. Mode is per-tenant config (`tenants.isolation_mode`).

### 6.1 Pool (Compliance tier — shared schema, RLS)

Every row carries `tenant_id UUID NOT NULL`. Every table has a `tenant_isolation` policy that reads the session variable `app.tenant_id`.

```sql
-- Migration 0001_enable_rls.sql
CREATE TABLE metric_observations (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id      UUID NOT NULL,
  metric_id      UUID NOT NULL,
  facility_id    UUID,
  fy             SMALLINT NOT NULL,
  period         DATERANGE NOT NULL,
  value_numeric  NUMERIC(28,6),
  value_unit     TEXT,
  source_kind    TEXT NOT NULL CHECK (source_kind IN ('ai','manual','api','erp')),
  evidence_id    UUID,
  confidence     NUMERIC(5,4),
  approved_at    TIMESTAMPTZ,
  approved_by    UUID,
  created_at     TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (id, tenant_id)
) PARTITION BY HASH (tenant_id);

ALTER TABLE metric_observations ENABLE ROW LEVEL SECURITY;
ALTER TABLE metric_observations FORCE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON metric_observations
  USING (tenant_id::text = current_setting('app.tenant_id', true))
  WITH CHECK (tenant_id::text = current_setting('app.tenant_id', true));

-- Service role bypasses for scheduled jobs (e.g. cross-tenant benchmarking)
CREATE ROLE app_service BYPASSRLS;
GRANT app_service TO brsr_workflow;
```

The NestJS `TenantInterceptor` runs `SET LOCAL app.tenant_id = $1` inside every request transaction so RLS is enforced even if a developer forgets a `WHERE tenant_id = ?` clause.

### 6.2 Silo (Enterprise tier — schema-per-tenant)

- Each tenant gets its own Postgres schema `tenant_<uuid8>`.
- Prisma is instantiated per-request with `?schema=tenant_<id>` search path.
- Migrations applied via a fan-out script over `pg_namespace` LIKE `tenant_%`.
- Storage prefix: `s3://brsr-evidence-vault-prod/t/<id>/`.

### 6.3 Single-tenant (Group tier — dedicated cluster)

- Dedicated Aurora cluster, dedicated Redis, dedicated Qdrant collection.
- Same EKS, dedicated namespace `tenant-<id>` with NetworkPolicy denying cross-namespace traffic except via the mesh.
- Customer KMS CMK accepted.

### 6.4 Single-tenant VPC (Listed Premium — dedicated VPC + region)

- Whole VPC dedicated. Optional BYO region (Mumbai default, Hyderabad alternative).
- Customer-managed Vault (Transit secrets engine for envelope encryption).
- Optional air-gapped deployment with mirrored Anthropic Bedrock endpoint.

### 6.5 Per-tenant Qdrant collections

```python
def collection_for(tenant: Tenant) -> str:
    if tenant.tier in ("compliance",):
        return "pool_shared_v3"  # filtered by payload {tenant_id: ...}
    if tenant.tier in ("enterprise",):
        return f"silo_{tenant.id.hex[:8]}_v3"
    return f"dedicated_{tenant.id.hex}_v3"
```

Pool collection uses Qdrant's payload index on `tenant_id` and enforces a `must` filter in every query. We never trust client-supplied filter alone — the server-side `QdrantClient.with_tenant(t)` factory always injects the filter.

### 6.6 S3 prefix scoping

IAM role per service is constrained:

```json
{
  "Effect": "Allow",
  "Action": ["s3:GetObject","s3:PutObject"],
  "Resource": "arn:aws:s3:::brsr-evidence-vault-prod/t/${aws:PrincipalTag/tenantId}/*",
  "Condition": {
    "StringEquals": {
      "aws:PrincipalTag/tenantId": "${s3:ExistingObjectTag/tenantId}"
    }
  }
}
```

The session token is generated with `sts:AssumeRoleWithWebIdentity` using the user's JWT, attaching `tenantId` as a principal tag.

### 7.4 Audit log + hash chain

```
audit_events {
  id            UUID
  tenant_id     UUID
  ts            TIMESTAMPTZ
  actor_id      UUID
  action        TEXT   -- e.g. metric.approve
  subject_kind  TEXT
  subject_id    UUID
  payload       JSONB
  prev_hash     CHAR(64)
  hash          CHAR(64)  -- SHA-256(prev_hash || canonicalise(payload) || ts)
}
```

A background job runs every 5 minutes computing `hash` for new rows; a daily job replicates the WORM-stamped chain to an S3 bucket with Object Lock (Compliance mode, 10y retention). The chain head hash is signed nightly with an HSM-stored Ed25519 key and posted to a public bulletin (optionally a blockchain notary for highest-tier customers).

### 7.5 PII handling

- PII columns (`users.email`, `suppliers.contact_phone`, `materiality_respondents.email`) live in dedicated tables, marked with `@pii` in the Prisma schema; the `PrismaPiiExtension` field-level encrypts them on write and decrypts on read.
- Right-to-erasure: a tenant admin can issue a redaction event; downstream tables hold a `subject_redacted_at` foreign key reference. Aggregates over redacted rows continue to work — only identifying fields are tombstoned.

## 8. AI Architecture

### 8.1 Five agents

| Agent | Model | Job | Typical latency |
| :--- | :--- | :--- | :--- |
| Extraction | claude-opus-4-7 | Read a document, return typed metric observations with bbox citations | 6 to 22 s |
| Classification | claude-haiku-4 | Tag a document by metric family (electricity, water, payroll, etc.) | 200 ms to 1 s |
| Anomaly | residual model + claude-opus-4-7 judge | Flag year-on-year, facility-vs-peer, period jumps | 3 s |
| Benchmark | claude-sonnet-4-7 + ClickHouse | Produce peer percentiles + narrative | 2 to 6 s |
| Narrative | claude-sonnet-4-7 | Draft BRSR narrative sections, variance commentary | 4 to 14 s |

All agents share the tool registry: `rag.search`, `sql.query` (read-only views), `calc.run`, `metric.lookup`, `file.cite`. The supervisor enforces a maximum of 8 tool calls per turn to bound cost.

### 8.3 RAG architecture

```
[Doc PDF] -> [Unstructured 0.16] -> [Layout-aware chunker] -> [voyage-3-large embed] -> [Qdrant upsert]
                                              |
                                              v
                                       [chunks table in Postgres,
                                        with bbox + page + parent]

  Query path:
  [user q] -> [HyDE expansion] -> [vector search top-50] -> [BM25 OpenSearch top-50]
              -> [reciprocal-rank fusion] -> [bge-reranker-v2-m3 top-10]
              -> [Opus 4.7 with cite-back schema] -> [strip uncited claims]
```

Chunks are 480 tokens with 80 overlap; tables are extracted separately and stored as both text + structured JSON; figures get an OCR + caption pass.

### 8.4 Hallucination prevention layers (5)

1. **Cite-back schema**: response JSON schema requires each metric value to include `citations: {doc_id, page, bbox, snippet}`. Pydantic strict validation. No citation, no answer.
2. **Citation verifier**: post-process step opens the cited page, regex-checks the snippet appears within the bbox.
3. **Cross-source check**: if the same metric is found in two documents, both must be cited; mismatch raises an Anomaly event.
4. **Judge pass (LLM-as-judge)**: a separate Opus call grades the extraction on a 1 to 5 rubric; scores below 4 route to HITL no matter the model confidence.
5. **Confidence floor**: anything < 0.7 goes to HITL queue, never to autocommit. Anything < 0.4 is rejected and the user is told the document needs better quality.

### 8.5 HITL flow

```
Extraction completes
       |
       v
   conf >= 0.95? --yes--> autocommit (still produces audit_event)
       |no
       v
   conf >= 0.7? --yes--> HITL queue (Reviewer role) -> approve / edit / reject
       |no
       v
   conf >= 0.4? --yes--> HITL queue with "low confidence" badge -> approve / edit / reject
       |no
       v
   surface error: "we could not extract reliably, please upload a cleaner doc"
```

Reviewer UI shows: PDF with bbox highlight, extracted JSON, judge rationale, 3 nearest prior approved extractions for the same metric/facility, suggested unit conversion.

### 8.6 Confidence scoring (5-component composite)

```
final_confidence = w1 * model_logprob_geomean
                 + w2 * judge_score / 5
                 + w3 * cite_back_validity
                 + w4 * unit_plausibility
                 + w5 * cross_doc_agreement

weights: w1=0.30, w2=0.25, w3=0.20, w4=0.10, w5=0.15
```

`unit_plausibility` checks the value against per-metric prior distributions (e.g. a facility electricity bill should be 1e3 to 1e8 kWh/month). `cross_doc_agreement` is 1.0 if no conflicting evidence, 0.0 if direct conflict.

## 10. Failure Modes and Recovery

### 10.1 Per-service degradation

| Service down | User-visible effect | Mitigation |
| :--- | :--- | :--- |
| ai-engine | Extraction jobs queue, dashboard shows "AI offline"; manual entry still works | Auto retry, page on > 5 min queue stall |
| copilot | Copilot chat unavailable; banner shown | All other features unaffected |
| Postgres writer | All mutations fail with retry-after; reads continue from replica | RDS failover in < 30 s |
| Redis | Sessions invalidated, idempotency disabled (logged warning), HITL queue paused | Multi-AZ failover, < 30 s |
| Qdrant | RAG returns BM25-only results | Auto fallback in tool registry |
| OpenSearch | RAG returns vector-only | Auto fallback |
| S3 | Uploads fail; reads from CloudFront cache continue | Multi-region replication for evidence-vault |
| Kafka | Outbox piles up in Postgres; no event-driven processing | Outbox is durable; backlog drains on recovery |

### 10.2 Backup strategy

- **Postgres**: Aurora continuous backup, point-in-time recovery to any second in last 35 days. Snapshot daily, replicated cross-region (ap-south-1 to ap-southeast-1 for DR). Logical `pg_dump` per tenant weekly to S3 evidence-vault under `/backups/`.
- **Qdrant**: snapshot daily, replicated cross-region; vector data is reproducible from chunks table + embeddings, used as secondary recovery path.
- **S3**: cross-region replication on, versioning on, Object Lock Compliance mode for evidence-vault (10 y), Governance mode for reports (7 y).
- **ClickHouse**: managed daily backups, 30 d retention.

### 10.3 DR plan

| Tier | RTO | RPO |
| :--- | :--- | :--- |
| Pool / Compliance | 4 h | 5 min |
| Enterprise (Silo) | 2 h | 1 min |
| Group | 1 h | 30 s |
| Listed Premium | 30 min | 0 (sync replica in second AZ + warm standby in alt region) |

Game-day exercise quarterly: simulated region failure, runbook drill, full restore validated against assurance-snapshot hash chain.
