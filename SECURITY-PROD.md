# BRSR AI Platform v2 — Production Security Checklist (Hostinger VPS)

This is the operational security checklist for running the BRSR platform
in production on a single Hostinger KVM 4 VPS. It is intentionally short
and actionable. See `SECURITY.md` for the broader application security
architecture (RBAC, audit log, signed evidence, etc.).

---

## 1. Network exposure

Only Caddy is reachable from the public internet. Verified with:

```bash
sudo ufw status verbose
```

Expected:

| Port | Proto | Purpose |
|------|-------|---------|
| 22   | tcp   | SSH (limit by IP if you can — `ufw allow from <office-ip> to any port 22`) |
| 80   | tcp   | HTTP (Caddy — redirects to 443) |
| 443  | tcp   | HTTPS (Caddy) |
| 443  | udp   | HTTP/3 (Caddy) |

Everything else MUST be blocked:

- 4000  (NestJS API)        — internal only, exposed via `expose:`, not `ports:`
- 8100  (AI engine)          — internal only
- 3000  (Next.js)            — internal only
- 5432  (Postgres)           — internal only, no `ports:` mapping
- 6379  (Redis)              — internal only
- 6333  (Qdrant)             — internal only
- 9000/9001 (MinIO + console) — internal only; access via SSH tunnel for admin

If you accidentally added `ports: 5432:5432` to postgres, `ufw` will NOT
save you — Docker bypasses ufw with iptables. Audit `docker compose -f
docker-compose.prod.yml ps` and confirm only Caddy shows host port
mappings.

## 2. SSH hardening

The bootstrap script enables `fail2ban`. Additionally:

1. Disable password auth once your SSH key works:
   ```bash
   sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
   sudo sed -i 's/^#\?PermitRootLogin.*/PermitRootLogin prohibit-password/' /etc/ssh/sshd_config
   sudo systemctl reload sshd
   ```
2. Optionally move SSH off port 22 (security through obscurity, but it
   does cut log noise).
3. Add the `brsr` user's key to `/home/brsr/.ssh/authorized_keys`; from
   then on, do not SSH in as root.

## 3. Secrets management

| Secret                       | Generated by                    | Rotation cadence |
|------------------------------|---------------------------------|------------------|
| `DB_PASSWORD`                | `openssl rand -hex 32`          | Annual or on compromise |
| `JWT_SECRET`                 | `openssl rand -hex 32`          | Annual; rotation invalidates all sessions |
| `NEXTAUTH_SECRET`            | `openssl rand -hex 32`          | Annual |
| `INTERNAL_CALLBACK_SECRET`   | `openssl rand -hex 32`          | Annual |
| `MINIO_ROOT_PASSWORD`        | `openssl rand -hex 16`          | Annual |
| `OPENAI_API_KEY`             | OpenAI dashboard                | Quarterly; immediately on suspected leak |

Rules:

- `.env` is **not** in git (covered by `.gitignore`).
- Permissions: `chmod 600 .env` and owned by the `brsr` user.
- Never paste secrets into Slack, Jira, or commit messages.
- Set an OpenAI usage cap on the platform dashboard
  (`Settings → Limits → Hard limit`) — protects you from a runaway
  extraction loop or a leaked key.

### Rotating OpenAI key

1. Create new key on OpenAI dashboard.
2. `nano .env` → replace `OPENAI_API_KEY=…`.
3. `docker compose -f docker-compose.prod.yml up -d --force-recreate ai-engine api`
4. Revoke the old key on OpenAI.

## 4. TLS

Caddy auto-provisions and renews Let's Encrypt certs for the value of
`DOMAIN`. Manual checks:

```bash
curl -I https://<DOMAIN>
docker compose -f docker-compose.prod.yml logs caddy | grep -i 'certificate obtained'
```

HSTS is enforced (`max-age=31536000; includeSubDomains; preload`). Do
not enable preload until you are certain you will keep HTTPS forever on
the apex domain.

---

## 5. MinIO

- Console (port 9001) is not exposed publicly. To access:
  ```bash
  ssh -L 9001:127.0.0.1:9001 brsr@VPS_IP
  # then visit http://localhost:9001 in your browser
  ```
- `brsr-evidence` bucket is created with **object lock** + **versioning**
  with a default 365-day COMPLIANCE retention. This means once written,
  evidence files cannot be deleted or overwritten by anyone — including
  the root user — for at least 365 days. This is the audit trail your
  assurance partner will rely on.
- `brsr-reports` is versioned but not locked (operators may need to
  regenerate reports).
- All inter-service S3 traffic stays on the docker `brsr` network —
  never traverses the public internet.

---

## 6. Database

- Postgres is **not** reachable from the public internet.
- The `brsr` Postgres role is the application role; it is also the
  superuser inside the container, which is acceptable for a
  single-tenant VPS but should be split when migrating to managed RDS.
- `pg_stat_statements` is enabled — useful for finding slow queries
  but it logs query text. Treat the Postgres data volume as PII.
- Daily encrypted backups go to MinIO `brsr-backups` (see `backup.sh`).
- Backup encryption: MinIO server-side encryption is enabled by default
  on `brsr-backups`. For extra safety pipe `pg_dump | gpg --symmetric`
  in `backup.sh` and store the passphrase in your password manager.

---

## 7. Container hardening

Each application image:

- Runs as a **non-root user** (`nestjs`, `aiengine`, `nextjs`).
- Uses `tini` as PID 1 so SIGTERM is propagated cleanly (no zombie
  processes during rolling restarts).
- Has a `HEALTHCHECK` — Docker restarts unhealthy containers via
  `restart: unless-stopped`.
- Has a `mem_limit` so a single misbehaving container cannot OOM the
  whole VPS.

## 8. OS update strategy

| What                                | Cadence | Tool |
|-------------------------------------|---------|------|
| Unattended security upgrades        | daily   | `unattended-upgrades` (enabled by bootstrap) |
| `apt-get update && apt-get upgrade` | weekly  | manual or cron |
| `docker compose pull` for base images | weekly | `docker compose -f docker-compose.prod.yml pull && ./scripts/deploy.sh` |
| Application code (`git pull`)       | per release | `./scripts/deploy.sh` |
| Kernel reboots                      | monthly | `sudo reboot` during a maintenance window |

Recommended weekly cron (run as root):

```cron
0 4 * * 0  apt-get update -y && apt-get upgrade -y && docker system prune -af --volumes=false
```

## 9. Logging & retention

- Caddy access logs: 100 MB rolling, 5 files kept (~500 MB), 30 days
  via `roll_keep_for 720h`. Sufficient for incident forensics on a
  single VPS.
- Application logs: written to stdout, captured by Docker's
  `json-file` driver. Cap log file growth so disk doesn't fill:

  Add to `/etc/docker/daemon.json` (then `sudo systemctl restart docker`):
  ```json
  {
    "log-driver": "json-file",
    "log-opts": { "max-size": "100m", "max-file": "5" }
  }
  ```
- Export logs to an external SIEM (Better Stack, Loki, CloudWatch) as
  you grow — a compromised VPS makes local logs untrustworthy.

## 10. Incident response runbook

### Symptom: extraction is failing / strange model output
1. `docker compose -f docker-compose.prod.yml logs ai-engine --tail 200`
2. Check OpenAI dashboard for anomalous usage / quota breach.
3. If suspected key leak: rotate immediately (section 3).

### Symptom: suspected breach of a container
1. Snapshot state for forensics:
   ```bash
   docker commit brsr-<service> brsr-<service>-incident-$(date +%s)
   docker compose -f docker-compose.prod.yml logs <service> > /tmp/incident-<service>.log
   ```
2. Stop the container: `docker compose stop <service>`.
3. Rotate **all** secrets in section 3.
4. Restore data from the most recent known-good MinIO backup:
   ```bash
   docker compose run --rm \
     -v /home/brsr/brsr-v2/data/backups:/restore \
     postgres bash -c "gunzip -c /restore/brsr-pg-<TS>.sql.gz | psql -h postgres -U brsr -d brsr"
   ```
5. Forensic review of Caddy access logs for the attacker IP, then add
   to `ufw` deny list.

### Symptom: VPS unreachable / Hostinger outage
1. Check status.hostinger.com.
2. If a hard reboot is needed: Hostinger panel → **VPS → Manage → Reboot**.
3. Containers auto-start due to `restart: unless-stopped`.

### Symptom: data corruption
1. Stop API + ai-engine + web (keep Postgres up):
   ```bash
   docker compose -f docker-compose.prod.yml stop api ai-engine web caddy
   ```
2. Restore most recent good backup from MinIO `brsr-backups/postgres/`.
3. Restart application services.

## 11. Pre-production checklist

Before pointing a paying client at the platform:

- [ ] `DOMAIN` set to a real, DNS-resolving hostname.
- [ ] HTTPS works in a private browser window (valid Let's Encrypt cert).
- [ ] `OPENAI_API_KEY` works (`/api/v1/health/ai` returns OK).
- [ ] All 6 secrets in section 3 are unique, not `CHANGE_ME`, and not the
      example values.
- [ ] `chmod 600 .env`.
- [ ] SSH password auth disabled; root SSH disabled.
- [ ] `ufw status` shows only 22/80/443 open.
- [ ] `./scripts/backup.sh` succeeds end-to-end.
- [ ] Cron entry installed for nightly backup.
- [ ] OpenAI usage cap configured (hard limit + email alert).
- [ ] Hostinger account has 2FA.
- [ ] Disaster recovery: documented who has the .env, the SSH key, and
      the Hostinger login.
